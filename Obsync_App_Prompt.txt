The app is an enterprise-grade SQL Server object version control and GitHub sync tool.

Core product idea:
Build a Windows desktop app that connects to SQL Server, scripts database objects at very high speed, detects object changes compared with the previous GitHub upload, and commits/pushes only the latest changed SQL object scripts to GitHub. The app should be used by enterprise companies, DBAs, database developers, compliance teams, and engineering teams who want automatic GitHub version control for SQL Server database objects.

Core workflow:

1. User creates a Sync Job.
2. User selects SQL Server connection.
3. User selects one or more databases.
4. User chooses object types using simple presets: Recommended, Programmability Only, Full Schema, Custom.
5. User connects/selects GitHub repository.
6. User selects branch and destination folder path.
7. User chooses schedule: Manual, Hourly, Daily, Weekly, Custom Cron.
8. User reviews a clean summary.
9. User saves the job and can run immediately.
10. Background service runs the job on schedule.
11. Engine scripts SQL objects.
12. App normalizes script output.
13. App generates SHA-256 hash per object.
14. App compares with previous hash/local state and Git workspace.
15. Only changed, added, or deleted objects are written.
16. App commits changes to Git.
17. App pushes to GitHub.
18. App stores run history, object hashes, commit SHA, duration, and status in SQLite.
19. UI shows job status, change summary, commit link, logs, and next run.

Performance requirements:
This app must be extremely fast, even for very large databases with hundreds of thousands or millions of objects.

Design the engine to avoid N+1 database round trips.

Use:

* Bulk metadata queries
* Async SqlDataReader
* Streaming where possible
* Object inventory read in one/few queries
* Bulk module definition reads from sys.sql_modules
* Parallel processing by database/object type with bounded concurrency
* Async file writes
* Hash-based change detection
* Skip unchanged files
* Resume failed runs
* Cancellation tokens
* Retry handling for transient SQL/GitHub/Git failures
* Memory-safe processing
* Backpressure where needed, such as System.Threading.Channels or TPL Dataflow

Scripting strategy:
Use a hybrid scripting engine.

Fast path using raw SQL Server metadata:

* Schemas
* Stored procedures
* Views
* Functions
* Triggers
* Synonyms
* Sequences
* Tables where possible
* Indexes where possible
* Constraints where possible
* Users/roles/permissions where possible
* Extended properties where possible

Use sys.sql_modules.definition for:

* Stored procedures
* Views
* Functions
* Triggers

Use structured metadata queries for:

* Tables
* Columns
* Data types
* Identity columns
* Computed columns
* Defaults
* Primary keys
* Foreign keys
* Indexes
* Unique constraints
* Check constraints
* Extended properties

Use SMO fallback for complex or high-fidelity scripting cases:

* Complex table definitions
* Partition schemes/functions
* Full-text catalogs/indexes
* Assemblies
* Security edge cases
* Advanced SQL Server features
* Any object type where raw scripting is risky

Change detection:
Implement normalized script hashing.

For each object:

* Generate canonical script text.
* Normalize line endings.
* Normalize trailing spaces.
* Optionally normalize SET options/header noise.
* Generate SHA-256 hash.
* Compare with last stored hash.
* If changed, write file and stage for Git commit.
* If unchanged, skip.
* If object was dropped from SQL Server, optionally delete file and commit deletion.

Git/GitHub workflow:

1. Clone repository if local workspace does not exist.
2. Pull latest branch.
3. Ensure destination folder exists.
4. Script objects into deterministic folder structure.
5. Stage only changed/added/deleted files.
6. If no changes, do not commit.
7. If changes exist, create clean commit.
8. Push commit to GitHub.
9. Store commit SHA and show link in UI.

Commit message format:
[DatabaseName] SQL object changes from [ServerName] - yyyy-MM-dd HH:mm

Commit body should include:

* Server
* Database
* Job name
* Run ID
* Objects scanned
* Objects added
* Objects modified
* Objects deleted
* Duration
* List of changed files, capped if too many
