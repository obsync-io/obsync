# Obsync production-readiness audit — bug report

Audit date: 2026-07-15 · Baseline: `002afdf` (v0.8.2) · Environment: Windows 11, .NET SDK 10.0.302,
SQL Server 2025 (17.0.1125.2) local instance, git 2.50.1.

Method: six parallel fresh-context code audits (engine/change detection, scheduler/service,
Git/GitHub, security, UI wiring, failure handling/installer) plus an option-truthfulness sweep,
cross-checked against a runtime end-to-end harness (`tools/Obsync.E2E`) driving the real pipeline
against disposable SQL Server databases and local bare git remotes. Every fix below was verified by
a regression test and/or the E2E battery; "Evidence" cites the observable proof from this session.

Severity: **Critical** = whole runs fail or committed history corrupted for common inputs.
**High** = data loss / wrongful deletions / silent missed runs / permanent wedges. **Medium** =
wrong behavior with workarounds or misleading state. **Low** = polish, honesty, edge cases.

## Confirmed, not fixed (documented residual risks)

| ID | Sev | Component | Description | Mitigation / disposition |
|---|---|---|---|---|
| OPEN-01 | High | Git / engine | **Non-fast-forward wedge needs manual recovery.** If a push fails AND the remote branch advances externally, every later run warns "non-fast-forward" until a human reconciles the workspace clone. Verified live (E2E S08): the failure is loud, actionable, and never force-pushes or destroys the foreign commit — but it does not self-heal. Compounding risk: direct mode marks changes delivered at commit time, so deleting the wedged workspace (or a corrupt-clone re-clone in that state) silently loses the stranded changeset until each object next changes. | Loud + safe today; self-healing reconciliation (regenerate-on-reset) is designed but deliberately deferred — it changes delivery-gate semantics and needs its own careful test cycle. Documented in RELEASE_READINESS. |
| OPEN-02 | Medium | PR mode | Un-merged PR content is re-reported as "Modified" on every subsequent run (head branch recut from base; the rewrite is required so unmerged work stays in later PRs, but the reporting is noisy) and PR head branches accumulate on the remote. | Behavior is safe; documented. |
| OPEN-03 | Medium | Incremental | A table's inline permission/extended-property sections can lag under incremental scripting (grants/EPs don't bump `modify_date`). Verified on SQL 2025: index create/drop and trigger creation DO bump it; grants and extended properties do not. The consolidated `security/permissions/permissions.sql` captures grants every run. | Caveat documented in the option text; disable incremental for one run to capture. |
| OPEN-04 | Medium | App | No UI to enable/disable a job (the flag exists, is honored end-to-end incl. the new engine gate, but only job-import writes it). | README corrected; product decision for the owner. |
| OPEN-05 | Low | Scheduler | Hourly cadence with a maintenance window: reconcile refreshes NextRunAt with the raw Quartz fire (possibly in-window-skipped), so the displayed value can flap against the engine's window-adjusted stamp. | Wizard now blocks the fully-starved configs; display-only inconsistency remains for hourly+window. |
| OPEN-06 | Low | Locks | `JobRunLock.IsHeld` probes by briefly acquiring the lock (a tiny startup race window); an ACL failure on the locks folder reads as "already running". | Windows are tiny; failure mode is a skipped occurrence with a logged reason, never a duplicate run. |
| OPEN-07 | Low | Engine | `LocalExportPath` mirror receives changed files only — never deletions; it accumulates stale files for dropped objects. | Document as "changed files are copied here". |
| OPEN-08 | Low | CLI | `obsync run` bypasses the production-tag confirmation (app-only guard). | README now states the scope. |
| OPEN-09 | Low | Various | Git commit messages passed via argv could exceed the Windows command-line limit at extreme path counts; no per-git-command timeout (a blackholed connection holds the repo lock until cancellation/30-min bound); manual-proxy bypass list applies to Octokit only; a hand-edited custom `RemoteUrl` (GHES) splits git-vs-API hosts with no validation; job-import doesn't clamp out-of-range advanced values; no single-instance guard for the app (safe but second instance loses file logging); `MaxScriptChars` counts chars not UTF-8 bytes; `run.CommitUrl` is set before the push is known to succeed (404 until re-push); database snapshots would be enumerated by all-user-databases scope. | Individually small; none corrupts state. ## Incidental live finding (user's machine, disclosed)

During service testing, a console-hosted service instance briefly attached to the **real** data root
(see OBS-43) and executed a pending catch-up of the user's job "Local". It completed in 16 s as
**NoChanges** (no commit, no push; one benign history row + audit event). The revealing part: a
catch-up from **2026-07-11** was still pending although the installed Obsync service has been
Running — which is exactly OBS-00's signature: the installed 0.8.2 service crashes on every plain
cron fire, so the job's hourly schedule has never actually executed. The installed service needs
the OBS-00 fix (upgrade) to start honoring schedules; its Windows Application event log should
contain the `Key trigger not found` errors as corroboration.
